## 三、10 条补充经验（分 3 层）

### L1：对 M5 canonical reference 的需求扩展（3 条，最高价值）

> 这 3 条直接决定 v1.5 Spec 引擎反推的质量上限。ADR-0006 已经定了**代码级**质量规范（缓存/定时/文件/审计），但**漏掉了业务建模级**规范。必须在 M5 canonical reference 详细设计文档落地时补上。

#### L1.1 容错设计范式写进 M5 canonical 需求文档

**AI 盲区（产品级）**：AI 写 canonical reference 时会默认采用"纯 CRUD + 字段校验不通过就抛 400 ProblemDetail"的模式。这是 AI 训练语料的中位数，**不是好设计**。

**实战经验**：佑海在位置批量导入场景中验证过"**引导用户就地修复，而不是拒绝后让用户离开当前流程**"的设计模式：

```
上传/提交数据
  ↓
预览（标记所有问题，不是遇到第一个就停）
  ↓
就地修复（不离开当前弹窗/页面）
  ↓
重新校验（自动，用同一份数据）
  ↓
确认提交
```

**关键设计要点**：
- 预览阶段所有行都展示，错误行标红 + **具体原因**（不是"数据无效"，而是"城市[qew]在该国家下不存在"）
- 修复阶段只展示**可修复**的问题（缺失的关联数据），不展示必须回源头改的问题（格式错误、必填为空）
- 所有字段可编辑 + 预填原始值，用户可能需要纠正（英文名填到了中文名字段）
- 有依赖关系时分阶段创建（先建省拿到 ID 再建城市），但对用户是一次点击
- 批量操作一次请求，不要逐条发
- 最终只导入 VALID 行，单条失败不影响其他行

**对 meta-build 的致命影响**：

> **Spec 引擎（v1.5+）的质量上限 = M5 canonical reference 的质量上限。**

`business-order` 目前的 M5 设计是"主从表 + 状态机（DRAFT→SUBMITTED→APPROVED→REJECTED→CANCELLED）"——如果只做这种纯 CRUD 骨架，未来 Spec 引擎反推时学不到任何产品意识。使用者用 Spec 生成的**所有**模块都会是"冷冰冰的校验拒绝器"。

**落地位置**：
- 新建 `docs/specs/canonical-reference-extensions.md`，或者在 M4 末写 `canonical-reference-features.md` 时补进去
- `business-order` 或 `business-approval` 必须**至少有一处**"预览 → 就地修复 → 重新校验 → 确认"的完整交互样本
- 在 M5 对应模块的 ADR 里明确记录："**为什么采用容错范式而不是纯 CRUD 拒绝模式**"
- 前端 `@mb/ui-patterns` 的 NxDrawer 和 NxForm 在 M3 设计时必须预留"预填 + 逐行错误标注 + 就地修复"能力，不能等到 M5 才加

#### L1.2 M5 必须有一个"第三方系统集成"的 canonical reference

**结构性缺口**：附录 B.4 当前的 M5 canonical 是 **3 个纯内部业务**：`business-notice` / `business-order` / `business-approval`。整个 backend-architecture.md 里**没有任何"与外部系统集成"的代码样本**。

真实 SaaS/ERP 项目里，Webhook 接收、OAuth 授权、API 限流、状态延迟同步这些事情是绕不开的。但 Spec 引擎 v1.5 如果只看过内部业务样本，它反推出的模板就**完全无法覆盖对外集成场景**。

**nx-parent-shopify 的完整反面教材**：

```java
// /tmp/nx-parent-shopify/nx-adapter/adapter-shopify/rest/ShopifyController.java:76-137
// 同步接收 webhook，立即转 MQ 异步处理 — 这部分对
@PostMapping("/webhooks")
public void webhooks(@RequestHeader Map<String, String> headers, @RequestBody Object payload) {
    String topic = headers.get("x-shopify-topic");
    String shopDomain = headers.get("x-shopify-shop-domain");
    // ❌ 缺失：没有校验 x-shopify-hmac-sha256 签名（任何人都可以伪造 webhook）
    // ❌ 缺失：没有 OAuth 2.0 授权码兑换和 token 刷新逻辑
    // ❌ 缺失：MQ 消费失败后的死信队列处理
    producer.send(message);
    return; // 立即返回
}
```

**对 meta-build 的致命影响**：Spec 引擎反推质量 = M5 样本质量。如果 M5 没有集成样本，v1.5 Spec 生成的业务模块**全部只能做内部 CRUD**。

**落地建议**：

在附录 B.4 的 M5 canonical reference 清单里**新增第 4 个模块**：

```
| business-integration | M5 | 第三方系统集成范式 |
```

或者把现有的 `business-order` 扩展为"**订单 + 外部 Webhook 同步**"场景。这个样本**必须**覆盖下面 6 个要点（都是 AI 不会主动想到的）：

1. **HMAC-SHA256 签名验证**：作为 Interceptor/Filter 在 Controller 之前执行，验证失败直接 401。不能信赖业务代码调用验证函数
2. **Webhook 快速 ACK**：接口必须 < 3 秒返回，所有业务逻辑异步处理（Spring 原生事件 + `@Async`，v1 不引 MQ）
3. **幂等 key 的原子写入**：见 L2.1
4. **OAuth 2.0 完整流程**：授权码兑换 + access_token 刷新 + 按租户存储（映射到 `tenant_id`）
5. **API 限流 + 重试**：处理 429 + Retry-After header，指数退避
6. **定时补偿轮询**：见 L2.3

**落地位置**：
- `backend-architecture.md` 附录 B.4 增加 `business-integration` 条目
- M5 详细设计文档里写明"这是 meta-build 第三方集成能力的 canonical"
- 在 `docs/future-capabilities.md`（M4 结束时补）里记录：v1.5 Spec 引擎必须能反推出对外集成模板

#### L1.3 business-order 数据模型必须按 N:M 关系建模

**AI 盲区（建模级）**：AI 默认会把"订单 → 发货 → 快递单"建模成 **1:1 链式关系**。这在纯理论里看起来很干净，但**任何真实第三方系统**（Shopify / Amazon / 独立站 / 物流平台）都是 N:M：

- 一个订单可能拆成 N 个 fulfillment（多仓 / 多物流商 / 分批发货）
- 一个 fulfillment 可能合并 M 个订单（多件合并运输）
- 一个订单行可能部分发货 / 部分退款 / 部分换货

**实战教训**：nx-parent-shopify 的开发记录明确把这列为"方案问题"（"一个订单多个包裹、多个订单一个包裹、时序问题"），但代码里**没有真正解决**——只是识别了问题。项目能跑是因为业务场景单一。

**对 meta-build 的致命影响**：附录 B.4 当前对 `business-order` 的描述是"主从表 + 状态机"——默认就是 **1 订单 + N 订单行** 的经典 1:N 关系，没有考虑多 fulfillment / 多 shipment 的场景。如果 M5 按 1:1/1:N 建模，Spec 引擎 v1.5 反推时会把**"1:1 链式关系"固化为默认模式**。将来使用者接入任何真实物流/电商系统都会大返工——不是改几个字段的问题，是要改数据模型、状态机和 UI 的问题。

**落地要求**：

M5 `business-order` 的数据模型必须从第一天就按 **4 级 N:M** 建模：

```
order（订单头）
  ↓ 1:N
order_item（订单行）
  ↓ N:M
fulfillment（履约单）—— 一个 order_item 可以拆到多个 fulfillment，一个 fulfillment 可以合并多个 order_item
  ↓ 1:N
shipment（物流单）
```

**关键表设计**：
- 需要 `order_item_fulfillment` 关联表（多对多）
- `fulfillment.status` 和 `order.status` 是独立的状态机，不能联动
- `order.fulfillment_status` 是**派生字段**（由所有 fulfillment 状态聚合得出），不能直接写
- 状态转换必须有明确的不变量（例如"所有 fulfillment 都 delivered 才能标记 order 为 completed"）

**落地位置**：
- 附录 B.4 的 `business-order` 描述扩展为："**4 级 N:M 建模**（order → order_item → fulfillment → shipment）+ 状态机"
- M5 详细设计必须包含这个 ER 图
- ADR 记录"**为什么即使 v1 看起来用不到 N:M，也要这样建模**"（理由：Spec 引擎反推质量）

### L2：后端 AI 盲区的规则补充（3 条）

> 这 3 条补充到 `backend-architecture.md` 第 14 章反面教材清单。都是 ArchUnit 能部分拦截但需要明确规则的 AI 盲区。

#### L2.1 幂等必须是原子操作，禁止"先查后写"

**AI 盲区**：AI 写幂等保护的默认代码模式是：

```java
// ❌ 反模式：先查后写
if (redis.get(key) != null) return;  // step 1. 查
// ... 业务处理
db.insert(record);                    // step 2. 落库
redis.set(key, payload);              // step 3. 写标记
```

三步之间任何网络闪断、JVM GC、进程崩溃都会破坏幂等。同一条消息的两次投递可能都通过 step 1 的检查。

**实战教训**：nx-parent-shopify 的 `ShopifyWebhookService.java:69-83` **就是这个错误模式的活样本**：先 `redisUtil.hget` 查，再 `redisUtil.hset` 写，中间还夹着 DB 操作。Redis key 还没 TTL——会永久堆积。

**对 meta-build 的影响**：meta-build 的 HTTP POST 接口（创建订单、提交审批、支付回调）都会遇到重复请求。当前 `backend-architecture.md` 只在 7.9 讲了缓存穿透/雪崩/击穿，**没讲幂等**。canonical reference 如果不给出正确模板，AI 默认会写成"先查后写"。

**可执行规则（补进反面教材清单）**：

| # | 反面教材 | 防御机制 | 位置 | 验证方式 |
|---|---------|---------|------|---------|
| 16 | **反幂等"先查后写"**（Redis/DB 非原子保护在高并发下失效） | 统一用 `idempotency_keys` 表 + unique constraint 或 Redis `SET NX EX` | 新增 7.10 节 | `IdempotencyRuleTest`（grep 禁止"先 get 后 set"模式） |

**具体 MUST**：

- MUST NOT 用"先 `get` 后 `set`"实现幂等
- MUST 从以下三种中选一种：
  - **数据库 unique constraint**（推荐）：`idempotency_keys` 表的 `key` 字段 unique，`INSERT` 冲突即视为重复
  - **Redis 原子操作**：`SET key value NX EX <ttl>`（`NX` 保证不存在时才设置）
  - **乐观更新**：`UPDATE idempotency_keys SET processed=true, result=? WHERE key=? AND processed=false`，根据 affected rows 判断是否首次
- 幂等 key 存储必须有 **TTL**（建议 7-30 天），不能永久堆积
- 存储只存**哈希或状态**，不存完整 payload（节省空间 + 避免敏感信息泄漏）

**建议的 idempotency_keys 表**：

```sql
CREATE TABLE sys_idempotency_keys (
    key           VARCHAR(128) PRIMARY KEY,  -- 请求唯一 key（由客户端提供或服务端从幂等 header 提取）
    tenant_id     BIGINT       NOT NULL DEFAULT 0,
    action        VARCHAR(128) NOT NULL,     -- 业务动作（如 "order.create"）
    result_hash   VARCHAR(64),                -- 首次执行结果的哈希（用于重复请求时校验一致性）
    status        SMALLINT     NOT NULL,     -- 0=processing, 1=completed, 2=failed
    expires_at    TIMESTAMP WITH TIME ZONE NOT NULL,  -- 过期时间
    created_at    TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

**落地位置**：
- `backend-architecture.md` 新增 **7.10 幂等处理规范**
- 第 14 章反面教材清单增加第 16 条
- canonical reference `business-order` 的创建接口必须演示正确幂等实现

#### L2.2 tenant_id 在 `@Async / @Scheduled / MQ` 场景必须显式声明

**AI 盲区**：AI 写多租户代码时会倾向于从 `CurrentUser.tenantId()` 读取。这在同步 HTTP 场景下能跑，但遇到**非 HTTP 场景**时会出问题：

- `@Scheduled` 定时任务：没有 HTTP 请求上下文，`CurrentUser.tenantId()` **抛 UnauthorizedException**
- `@Async` 异步方法：ThreadLocal 天然不跨线程传递
- Spring 事件的 `@Async` 消费者：同上
- 未来的 MQ 消费者（v1.5+）：同上

**实战教训**：nx-parent-shopify 的 `ShopifyController.java:84` → `ShopifyWebhookService.java:80-81` 的做法是**正确的**：`shopDomain` 从 header 提取后，作为**显式参数**层层透传，最终作为一等字段存入每条记录。但这是因为他们没有用 Security 上下文抽象——如果用了，AI 会默认尝试从上下文读。

**对 meta-build 的影响**：

- ADR-0005 的 `CurrentUser.tenantId()` 是一个**隐式上下文读取接口**
- 反面教材 #5"反 ThreadLocal 异步丢失"**只识别了 DataScopeContext 的问题**，没明确说 tenant_id 在非 HTTP 场景下的处理
- `AsyncConfig` 的增强"RequestContextHolder 传递"**只解决了 HTTP → @Async 的链路**，没解决 `@Scheduled` 的冷启动场景

**具体 MUST**：

- MUST NOT 在 `@Scheduled` / `@Async`（冷启动，非 HTTP 继承） / 未来的 MQ Consumer / 线程池 `submit()` 的代码里直接调用 `CurrentUser.tenantId()`
- MUST 在进入非 HTTP 边界时选择以下之一：
  - **显式参数透传**：定时任务遍历所有租户时，`tenantId` 作为方法参数传入
  - **`@WithTenant(tenantId = ...)` 注解**：明确声明代码块的租户上下文
  - **`TenantContext.runAs(tenantId, () -> { ... })`**：显式的作用域包装，退出时自动清理
- ArchUnit 规则：被 `@Scheduled` 标注的方法，方法体内（递归调用链）禁止直接调 `CurrentUser.tenantId()`，必须通过显式参数或 `TenantContext.runAs`
- 多租户定时任务的标准模板（必须写进 canonical reference）：

```java
@Scheduled(cron = "0 0 2 * * *")
@SchedulerLock(name = "dailyReportJob")
public void runForAllTenants() {
    // ✅ 显式遍历租户，而不是依赖上下文
    for (Long tenantId : tenantApi.listActiveTenantIds()) {
        TenantContext.runAs(tenantId, () -> {
            // 现在这个作用域内，CurrentUser.tenantId() 返回 tenantId
            dailyReportService.generate();
        });
    }
}
```

**落地位置**：
- `backend-architecture.md` 第 5.3 节（DataScope opt-out）扩展，补充"非 HTTP 场景的 tenant_id 显式声明"
- 8.6 CurrentUser 接口增加 `TenantContext.runAs(tenantId, Runnable)` 静态工具
- ArchUnit 规则 `TenantContextRule`：`@Scheduled` 方法禁止直接调 `CurrentUser.tenantId()`
- 反面教材 #5 的描述从"DataScopeContext 丢失"扩展为"**CurrentUser 和 DataScopeContext 在非 HTTP 场景下必须显式声明**"

#### L2.3 事件驱动必须假设最终一致，配"定时补偿"兜底

**AI 盲区（深层）**：AI 会学会"用事件驱动解耦 + `@TransactionalEventListener(AFTER_COMMIT)`"（这部分 2.4 已覆盖），但**不会主动想到另一层事实**：

> **事件发出那一刻，事件里"说的"关联数据在外部系统可能还没真正落库。**

你 `OrderSubmittedEvent` 到了 approval 模块，你查对应的 user，`User.department_id` 可能还是旧值——iam 模块可能还没 commit 完。或者对接 Shopify 的 fulfillment.created 事件到了，你查对应的 order，`order.fulfillment_status` 可能还是 `unfulfilled`，Shopify 的内部索引几秒后才更新。

**实战教训**：nx-parent-shopify 的 `DataSyncScheduler.java:104-109` 和开发记录第 31-33 行明确记录了这个坑："时序问题：创建 fulfillment 后，order 中的 fulfillment 状态更新"。他们的解法是**每 45 秒轮询一次**"有 fulfillment 但 order 状态没对齐"的订单。

**对 meta-build 的普遍意义**：不只是外部对接。meta-build 的 `business-approval` canonical 是"跨模块协调"（iam + order + notification + audit），一旦这几个模块有独立的事务边界，就会出现同样的问题——approval 事件发出时，order 的状态可能还没 commit 完，notification 读到的是旧状态。

**对当前文档的补充**：

- 2.4 `AFTER_COMMIT` 只解决了**"事务回滚但通知已发"的单机半问题**
- **跨模块最终一致性**完全没覆盖
- **定时补偿兜底模式**完全没覆盖

**具体 MUST**：

- MUST NOT 在事件消费者里假设"事件携带的关联 ID 能立即查到最新状态"
- MUST 为跨模块 / 跨系统的状态同步流程**配备定时补偿任务**：扫描"状态应该已更新但没更新"的记录，主动对齐
- MUST 在 canonical reference `business-approval` 的 ADR 里明确声明：哪些流程是强一致（同一事务）、哪些是最终一致（补偿兜底）
- M5 `business-approval` canonical 里必须包含**一个"定时补偿"的实现样本**

**建议的补偿模板**：

```java
/**
 * 审批流程的最终一致性补偿：
 * 扫描"订单已提交但审批记录未创建"的订单（可能因为事件丢失或消费失败），主动补齐。
 */
@Scheduled(cron = "0 */5 * * * *")  // 每 5 分钟
@SchedulerLock(name = "approvalCompensationJob")
public void compensate() {
    List<Long> orphanOrderIds = orderApi.findSubmittedWithoutApproval();
    for (Long orderId : orphanOrderIds) {
        log.warn("approval.compensation: order {} 缺失审批记录，补偿创建", orderId);
        approvalService.createForOrder(orderId);
    }
}
```

**落地位置**：
- `backend-architecture.md` 2.4 事件机制章节增加"**最终一致性与补偿模式**"小节
- 反面教材清单新增第 17 条："**反假设事件 = 强一致**"
- canonical reference `business-approval` 必须演示定时补偿样本

### L3：前端 + 全栈规则库的第一批内容（4 条）

#### L3.1 建立规则库的 frontmatter triggers 发现机制

**AI 盲区**：AI 开工前不知道"这个任务有哪些历史坑要避开"。CLAUDE.md 可以写原则，但规则多了会爆。必须有一个机制，让 AI 能根据当前任务的关键词**自动发现相关规则**。

**实战经验**：佑海系统的 `04-rulers/` 目录用 frontmatter `triggers: [关键词列表]` + `scope: [领域]`，CLAUDE.md 里规定"开工前必须扫 rulers/ 目录，按 triggers 匹配相关规则再动工"。这是一个**可扩展的知识复利机制**——规则越积越多，但 AI 每次只读相关的。

**对 meta-build 的必要性**：meta-build 的规则库会比任何单项目都大（因为它是"给陌生开发者的脚手架"），没有这个机制 CLAUDE.md 必爆。

**规则文件模板**（放在 `docs/rules/` 下，命名 `RULE-<领域>-<短描述>.md`）：

```markdown
---
title: 规则标题
triggers: [触发词1, 触发词2, ...]
scope: [frontend / backend / contract / theme / security / workflow / data-model]
severity: [must / should / avoid]
introduced-in: <ADR 编号或规则创建日期>
---

# 规则标题

**背景**：这个规则在什么场景下出现
**问题**：不遵守会出什么错
**规则**：正确的做法是什么（MUST / MUST NOT 语气）
**验证**：怎么判断是否遵守（可选：CI 规则、grep 命令、ArchUnit 检查）
**适用范围**：什么情况下生效
```

**CLAUDE.md 必须新增的段落**：

```markdown
## 开工前的规则检查（强制）

开始任何编码任务前，必须：
1. 从任务描述中提取关键词（例如"修改 Button 组件"→ 关键词：component, primitives, ui）
2. 扫 `docs/rules/` 下所有 Markdown 文件的 frontmatter，按 triggers 匹配
3. 读取所有匹配的规则文件，按 severity 分类（must 必须遵守，should 建议遵守，avoid 反模式警告）
4. 在开始编码前明确告知：本次任务命中了哪些规则

不允许跳过这一步，即使任务看起来简单。
```

**落地位置**：
- 新建 `docs/rules/` 目录
- 新建 `docs/rules/INDEX.md`（按 scope 分类的规则索引）
- CLAUDE.md 骨架写入"开工前的规则检查"段落

#### L3.2 前端 5 条"AI 易漏"的工程纪律（一次性建立）

这 5 条是**同一类问题**：AI 倾向于"改了局部就报告完成"，不会主动搜索整个代码库的调用方。必须在 M0 就一次性建立。

| # | 规则 | 核心要点 |
|---|------|---------|
| 1 | **改 L1/L2 组件必须覆盖所有渲染路径** | 修改 `@mb/ui-primitives/*` 或 `@mb/ui-patterns/*` 时必须先 `grep -r "<组件名"` 找出所有使用点。TypeScript 类型检查**接不住** className 字符串、data-testid、DOM 结构变化。 |
| 2 | **重命名前必须全局扫描** | 重命名导出符号 / 路由 path / 权限 code / i18n key / OpenAPI operationId 前必须 `grep -rn "旧名字"` 全库扫。TanStack Router 的 path 字符串、i18n key、Sa-Token 权限 code 都是**静态重构工具感知不到的动态字符串引用**。 |
| 3 | **调 `@mb/api-sdk` 前必须读完后端校验逻辑** | TypeScript 类型只管"字段存在和类型正确"，不管业务语义（"订单状态为 DRAFT 时才能修改 amount"这种规则 AI 完全不知道）。必须打开对应的后端 Controller/Service，读完 `@Valid` 注解、手写校验、业务前置条件。 |
| 4 | **显示层文本映射必须收口** | 枚举到显示文本的映射必须定义在独立工具文件里（例如 `formatOrderStatus(status)`），**禁止**在组件内联写 switch/case 或 `status === 'DRAFT' ? '草稿' : ...`。配合 i18n，映射工具返回 i18n key。Biome 规则：features 组件内禁止"状态码中文字符串"字面量。 |
| 5 | **Loading 必须覆盖所有触发异步的 UI 路径** | 按钮、筛选、翻页、排序、mutation、衍生请求——所有触发网络请求的操作都要有 loading。用 TanStack Query 的 `isFetching`（不是 `isLoading`，后者只在首次加载 true）。按钮在 mutation pending 时必须 disabled。`@mb/ui-patterns` 的 NxTable/NxDrawer 原生支持"所有子操作的 loading 态聚合"。 |

#### L3.3 通用组件禁止"万能口袋"（tempData 反面教训）

**AI 盲区**：AI 在设计通用组件时，遇到"有些业务需要额外字段"的诉求，会本能地加一个 `extraProps: Record<string, any>` 或 `tempData?: any` 当逃生舱。这个字段一旦加上就**永远去不掉**——没人知道谁在用它、存了什么、什么时候清。通用组件从此带上了业务耦合，API 稳定性承诺作废。

**实战反面教材**：佑海的 NxDrawer 为了"方便"加了 tempData 字段，结果具体存什么、给谁用、什么时候清全是业务代码决定。改 Drawer 时不敢动这个字段因为不知道谁在用；每个新业务都要学"怎么用 tempData 的黑话"。这条是佑海前端评估报告自己识别出来的。

**对 meta-build 的致命影响**：meta-build 规划明确要求 `@mb/ui-patterns` 的"API 对 features 暴露的 props 签名不可变"。一旦 NxDrawer 犯这个错，L3 就失去"稳定契约"的意义，5 层架构会从内部腐烂——而 5 层架构正是"千人千面"的物理基础。

**MUST NOT**：
- `@mb/ui-primitives` 和 `@mb/ui-patterns` 的 props 类型中**禁止**出现：
  - `any` / `unknown`（除非明确的类型守卫场景）
  - `Record<string, any>` / `Record<string, unknown>`
  - 名字叫 `extraProps` / `tempData` / `customData` / `meta` / `extras` 的字段
- 通用组件的所有 props 必须是**结构化的、显式的、可被 TypeScript 完整约束的**
- 业务诉求无法用现有 props 满足时：**(a)** 扩展一个显式命名的 prop 并进 ADR；**(b)** 用 render prop 或 slot 让业务代码自己写；**(c)** 不用通用组件，在 features 层单独实现。**永远不选"开口袋"**

**同样适用于后端**：通用 Service / BaseController 的参数类型也禁止 `Map<String, Object>` 当万能参数。

**落地位置**：
- `docs/rules/RULE-frontend-no-escape-hatch.md`（severity: must）
- M3 实施 `@mb/ui-patterns` 时的组件设计 Checklist 第一条
- Biome/ESLint 配置：在 `client/packages/ui-primitives` 和 `client/packages/ui-patterns` 的 tsconfig 禁用 `any`，加 lint 规则检查 props 接口中是否出现禁用的字段名

#### L3.4 安全权限"第一天启用"的反滑坡（补充反面教材 #7）

**AI 盲区**：AI 在搭建脚手架时会建议"为了开发方便先关掉 auth，等业务做完再开"。这句话听起来合理，实际上是一个**滑坡陷阱**。

**实战反面教材**：佑海系统安全评分 2.5/10，不是因为没做安全，而是 config 里有个 `auth.enable: false`——开发阶段临时关掉的，然后**再也没打开**。SQL 拦截器也因此形同虚设。这不是技术问题，是**文化滑坡**。

**对 meta-build 当前文档的补充**：

- 反面教材 #7"反硬编码敏感配置"**已经覆盖了** env var 无默认值、缺失启动失败
- **但没覆盖** "禁止存在 auth.enable / security.disabled / permission.bypass 这类全局开关"

**MUST**：
- **不允许**存在 `auth.enable` / `security.disabled` / `permission.bypass` 这类**全局开关**
- 开发环境需要绕过权限的场景必须通过以下方式之一：
  - 测试工具类 `MockCurrentUser` + `TestSecurityConfig`（**只在 test scope 可见**，ADR-0005 已定）
  - `@BypassDataScope` 方法级注解（必须显式标记，ArchUnit 规则限制只能在特定场景使用）
  - 开发环境种子数据包含超级管理员账号（**走正常登录流程**）
- CI 规则：`application-*.yml` 中**禁止**出现 `*.enable: false` / `*.disabled: true` 这类安全相关字段
- ArchUnit 规则：`@RequirePermission` 不允许出现永真表达式
- M1 脚手架搭起来的第一天，登录接口必须是真的 Sa-Token JWT，**不是 mock**

**落地位置**：
- `docs/rules/RULE-security-no-kill-switch.md`（severity: must）
- 反面教材 #7 的描述扩展为："**反硬编码敏感配置 + 反权限开关**"
- `CLAUDE.md` 骨架的 MUST NOT 段明确列出
- `.github/workflows/security.yml` 里加 grep 检查：任何 yml 中的 `auth.enable: false` / `security.disabled: true` 直接 fail
