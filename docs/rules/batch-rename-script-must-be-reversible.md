# 批量重命名脚本必须可回滚

## 规则
凡是批量重命名、批量迁移文件路径、批量改包名的脚本，**必须先设计回滚点和分阶段验证点**，不能直接在全量文件集上“一把梭”。

## Why

### 具体事件
2026-04-16 后端 DTO 批量重命名时，第一版 Python 重命名脚本路径计算出错，结果是：

- 旧文件已经被删
- 新文件没有完整生成
- 工作树进入“半迁移中间态”

最后不得不先用 `git restore` 把工作树救回，再修脚本重跑。

这类事故的根因不是“脚本写错一次”，而是：

> **把“适合脚本做”误当成“适合一次性全量执行”。**

批量脚本一旦出错，破坏范围是 O(n)，比手工改错更危险。

## How to apply
- 跑批量脚本前，先确保工作树干净，或者至少有一个明确可恢复的 checkpoint
- 先产出 rename/move 映射清单，再执行脚本
- 先拿 1-2 个文件做小样本验证，不要直接上全量
- 脚本执行后立刻做两类检查：
  - 残留 grep
  - 最小编译/测试验证
- 如果脚本执行失败，**先恢复工作树再修脚本**，不要在半迁移状态上继续叠改
- 批量迁移最好拆成两阶段：
  - 阶段 1：文件移动 + package/class 改名
  - 阶段 2：全仓引用替换
  每阶段结束都做一次验证
