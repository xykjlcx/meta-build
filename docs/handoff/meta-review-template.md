# 承接 Meta-Review 模板

> **触发场景**：新会话继承上个会话的交接文档 / 任务清单，或接手别人留下的 TODO 清单。
>
> **背景**：2026-04-18 meta-0023 后，再次确认"承接模式最大的认知盲点是默认每条任务都必要"。上个会话可能已经在过度工程，接手方若不质疑，下游继续放大。
>
> **硬性要求**：承接模式下，**未经 meta-review 的任务清单不得开始执行**。

---

## 为什么要 meta-review

交接文档是上个会话的产物，它**至少**有三类可能缺陷：

1. **伪问题任务**：上个会话把"技术现象"当成"必须解决的问题"，但当前阶段其实不需要（参见 meta-0023 / `docs/rules/over-engineering-trap.md`）
2. **过时决策**：交接后项目上下文变了（新 commit、新 ADR、新协议），清单里某些任务已经被环境抵消
3. **不对齐的优先级**：交接人当时的注意力是 X，但接手方看到的全局可能是 Y 优先

**meta-review 的目的不是否定交接文档，是把"默认执行"改成"默认质疑"。**

---

## 操作流程

### Step 1：读 handoff 文档原文（不要直接执行）

一次性全读完整份交接，先不做任何操作、不建任务列表。只看。

### Step 2：扫一遍上下文的"环境快照"

- `git log --oneline -20`
- `git status` + 当前未 push 的 commit
- `docs/rules/INDEX.md`（本项目的 pitfall / playbook 索引）
- `docs/adr/` 最近 3 份 ADR 标题
- `docs/handoff/v1-pre-prod-checklist.md`（已知延后清单）
- 交接文档里提到的关键 commit 的 `--stat`

这一步目的是校准"交接文档说的现状 vs 当前仓库实际现状"。交接文档可能写于几小时前，期间可能有其他 session 推过 commit。

### Step 3：对交接清单的每一条回答 4 个问题

**不允许跳过、不允许合并。每条都答。**

```markdown
## 任务：<原清单第 N 条描述>

### Q1：这件事在**当前阶段**影响有多大？
举具体数字 / 事实 / 监控数据。不允许"可能会""理论上""万一"。
如果你只能给"可能会"，说明这是伪问题信号。

### Q2：不做会怎样？
列最坏情况。如果最坏情况是"一个没被触发的风险"、"pre-prod 之前不影响"，
降级到 🟡。

### Q3：解决方案的复杂度 vs 问题严重度，比例合理吗?
解法引入的代码量 / 新依赖 / 新约束，和 Q1 说的影响相比是不是失衡?

### Q4：能登记到 pre-prod checklist 等真实数据再决定吗?
对"性能 / 可观测 / 安全加固"类任务，9/10 的答案是"能"。

### 分类：🔴 立即做 / 🟡 登记到 pre-prod / 🔵 撤销
只有 Q1-Q3 都明确答"是必须"才进 🔴。
任一条答不出具体数据，默认 🟡 或 🔵。
```

### Step 4：输出 meta-review 报告给用户

**不直接执行**。给用户一份清单，形如：

```markdown
# 承接 Meta-Review 报告

原清单 N 项，meta-review 后分类：

🔴 立即做（X 项）：
- <具体任务> — 理由：<Q1 的具体数据>
- ...

🟡 登记到 pre-prod（Y 项）：
- <任务> — 登记原因：当前 QPS < 1 / 无真实流量 / 无监控数据，待 Z 后评估
- ...

🔵 建议撤销（Z 项）：
- <任务> — 伪问题理由：<Q1-Q3 的 red flag>
- ...

请确认分类。确认后我只执行 🔴 部分，🟡 登记到 pre-prod checklist，🔵 不动。
```

### Step 5：等用户确认 → 只执行 🔴

收到明确确认之前不动代码。

## Red flag 速查（触发即降级）

下列表述出现在交接文档里，**相应任务默认不是 🔴**（除非有强数据反驳）：

| Red flag 短语 | 降级提示 |
|---|---|
| "上生产前应该..." | 🟡 pre-prod |
| "潜在的 XXX 风险" | 🟡 / 🔵 |
| "性能可能有问题" | 🟡 需 benchmark |
| "隐患 / 最佳实践 / 补强" | 🟡 / 🔵 |
| "顺手把 Y 也做了" | 🔵 非必须 |
| "既然已经做到这了，那..." | 🔵 sunk cost 谬误 |
| "为了一致性 / 完整性..." | 🟡 检查是否真必要 |

## 一个反面样本

**2026-04-18 会话承接**

上个会话的交接文档里列了 15 项任务。本次接手方（小灵犀）**没走本模板**，默认全部 🔴 执行，结果：

- 🔴 Actuator 独立端口改 prod profile：引入 89 个测试 context 失败，3 轮试错才回滚
- 🔴 CompositeTaskDecorator 重构成 ObjectProvider + @Order：没 benchmark 证据，引入自引用风险
- 🔴 补 `async_must_not_leak_mdc` 测试：补测试合理，但"线程池复用"场景无法保证
- 🔴 delivery_log 字段重构：在 meta-0023 撤销通知相关 ADR 的同时，还对 delivery_log 做了一次 schema 重构——这两件事之间的联系在交接里没讲清

**如果当时走了 meta-review**，这几项大概会被降级到 🟡（等真实流量 / benchmark 数据）。

此外，上个会话提议的"写 ADR-0023"是真 🔴（已在讨论中形成共识，不执行会 drift），这个确实该立即做。**meta-review 不是否定一切，是区分信号。**

## 维护约定

- 本模板是 pitfall 的产物，不是 playbook（不是"好的做事方式"，是"防止坏的做事方式"）
- 如果某次 meta-review 后 🔴 降级率 < 30%，可能是交接文档质量极高（少见）或 meta-review 太宽松（常见），偏向后者自检
- meta-review 报告**必须保留**在 `docs/handoff/<date>-meta-review.md`，未来追溯交接质量
