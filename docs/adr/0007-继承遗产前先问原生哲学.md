## 背景

### 触发事件：数据权限方案的 5 轮迭代

meta-build M0 阶段讨论"数据权限 opt-out"的实现方案，在原 backend-architecture.md §5.3（现 backend/05-security.md §7）连续经历了 5 轮方案迭代：

| 轮次 | 方案 | 问题 |
|-----|------|------|
| 第 1 轮 | 直接搬 nxboot 的 `DataScopeAspect + DataScopeContext` | opt-in 模式漏风险 |
| 第 2 轮 | 反转为 `DataScopedRepository` 基类 + `DataScopeVisitListener` **双保险** | 基类仍然是"子类必须调用 `applyDataScope()`"，本质还是半调子 opt-in |
| 第 3 轮 | 拆分 `DataScope` 类型到 `infra-security`，基类到 `infra-jooq`，**放宽 §1.5 允许 infra 子模块单向依赖** | 破坏 §1.5 硬约束，且冗余的 `DataScopeContext` ThreadLocal 仍然存在 |
| 第 4 轮（B''） | 把 `DataScope` 类型沉到 `mb-common`，去掉 `DataScopeContext`，基类依赖 `CurrentUser` | 基类仍然存在 |
| 第 5 轮（**方案 E**） | **砍掉 `DataScopedRepository` 基类**，`DataScopeVisitListener` 单点拦截 + `DataScopeRegistry` 集中声明 + 普通 Repository + `NO_RAW_SQL_FETCH` ArchUnit 兜底 | 达到终态 |

### 洋哥的质疑（前四轮之后）

第 4 轮完成时，洋哥问了两个问题——正是这两个问题把方案推到终态：

1. **"你觉得这是终极方案吗？"**
2. **"回到源头来想，是否我们一开始就应该这么设计呢？"**

这两个问题戳破了前四轮迭代的共同盲区：**我一直在优化"基类怎么变好"，但从没问"为什么需要基类"**。

### 根因分析

回看 5 轮方案，前四轮有一个共同假设：**`DataScopedRepository` 基类是必要的**。

这个假设**不是来自第一性原理**，而是来自 nxboot 的继承——nxboot 是 MyBatis-Plus 生态（`extends ServiceImpl<Mapper, Entity>` 是所有业务 Service 的强制范式），基类继承是 MyBatis-Plus 的默认心智模型。

但 meta-build 用的是 **jOOQ**，jOOQ 的设计哲学和 MyBatis-Plus **完全相反**：

| | MyBatis-Plus 生态 | jOOQ 生态 |
|---|---|---|
| **扩展范式** | **继承**（`extends ServiceImpl` / `extends BaseMapper`） | **组合 + 函数式**（`DSLContext` 作为参数传入，静态工具 + DSL 链式调用） |
| **Repository 形式** | 接口 + XML / 注解，或基类 + generic | 普通 Java 类 + `DSLContext` 字段 |
| **数据权限的社区标准** | Interceptor / Plugin + 基类方法 | **`VisitListener` 单点拦截**（jOOQ 官方文档 + Lukas Eder 博客 + Stack Overflow 主流答案） |
| **AOP 倾向** | 高（MyBatis 本身就大量用 AOP） | 低（jOOQ 鼓励显式写查询） |

**把 MyBatis-Plus 的"基类继承"范式搬到 jOOQ 世界，就是在两个不同哲学的生态之间搬运反模式**。前四轮方案都在修补这个反模式的症状（依赖反转、ThreadLocal 泛滥、元数据管理痛），而不是挑战根本假设。

第五轮（方案 E）砍掉基类后，所有前四轮的症状**同时消失**：

- 没有基类 → 没有继承耦合 → Repository 是普通类，AI 读代码无心智负担
- 没有基类 → 没有"子类调 `applyDataScope()`"的隐式约定 → VisitListener 是唯一拦截点，语义清晰
- 没有 `DataScopeContext` → 整个项目只剩 Sa-Token 一个 ThreadLocal，异步丢失问题收敛
- `infra-jooq` 不需要依赖 `infra-security` → §1.5 硬约束零放宽
- 所有"受保护表"在 `DataScopeRegistry` 一个文件集中声明 → 数据权限全景可 grep

## 决策

**作为 meta-build 的元方法论**：

> **从 nxboot（或任何遗产项目）借用组件时，在决定"如何集成"之前，必须先问两个问题：**
>
> 1. **这个组件的范式是否来自它所在生态的约定？**（例：MyBatis-Plus 的基类继承是该生态的约定）
> 2. **新技术栈的原生哲学对同一问题的标准答案是什么？**（例：jOOQ 的标准答案是 VisitListener，不是基类）
>
> **如果答案 1 和答案 2 不一致，优先采用新技术栈的原生哲学，而不是无条件继承遗产的实现。**

### 这条元规则的适用范围

| 决策场景 | 错误做法（继承惯性） | 正确做法（原生哲学） |
|---------|------------------|------------------|
| 数据权限 | nxboot：`DataScopeAspect` + 基类 → 机械搬运 | **jOOQ：`VisitListener` 单点**（方案 E） |
| 认证上下文 | nxboot：`SecurityUtils.getCurrentUser()` 静态方法 → 搬运 | **Sa-Token 原生：`StpUtil`，但通过 `CurrentUser` 门面隔离**（ADR-0005） |
| 模块边界 | nxboot：没有模块边界 → 沿用 | **Spring Boot 3 生态：Spring Modulith 或 Maven pom 隔离**（ADR-0003 选了后者） |
| SQL 构建 | MyBatis-Plus：XML / 注解 / 基类 → 搬运 | **jOOQ：类型安全 DSL + 普通类 Repository**（方案 E） |
| 分页 | MyBatis-Plus：`PageHelper` 切面 → 搬运 | **jOOQ：`Select.limit().offset()` 显式调用 + `PageResult<T>` 返回**（规划决策 9） |

### 推论：继承 nxboot 组件时的审查流程

每一条 "从 nxboot 借用 XXX" 的决策，**都必须在附录 A 借用清单里回答三个问题**：

1. **原生范式**：这个组件在 nxboot 里是基于什么生态（MyBatis-Plus / Spring Security / 其他）的约定？
2. **新生态答案**：meta-build 用的新技术栈（jOOQ / Sa-Token / etc.）对同一问题的原生答案是什么？
3. **改造策略**：直接搬、部分借用、**只借鉴概念完全重写**、**彻底放弃**。

如果选择"只借鉴概念完全重写"或"彻底放弃"，必须在改造点列里**显式说明原因**，避免未来有人看到 nxboot 的组件名以为是"平移过来的"。

### 元规则的底层直觉

**"新项目"和"老项目"的区别，不是技术栈列表，而是哲学基因**。继承老项目的"实现"容易，继承老项目的"哲学基因"致命。meta-build 的北极星是"给 AI 执行的契约"，AI 在每个生态里的熟练度都是基于**该生态的原生范式**训练的——让 AI 在 jOOQ 项目里写"extends DataScopedRepository"的代码，等于让它用一个不熟悉的、混合范式的模式工作，错误率必然高。

这条元规则其实是**对"AI 友好"北极星的具体化**：AI 友好 = 选择新技术栈的原生哲学，避免跨生态范式污染。

## 方案 E 的具体内容

参见 `docs/specs/backend/05-security.md` §7（方案 E，原 backend-architecture.md §5.3） 完整的代码骨架和设计文档。核心结构：

```
mb-common.security         ← 类型定义 + 公共抽象接口
├── CurrentUser            # 接口（方案 E 从 infra-security 挪到这里）
├── DataScope              # record
├── DataScopeType          # enum
└── BypassDataScope        # 注解

infra-jooq                 ← 数据权限的唯一归属
├── DataScopeRegistry      # 使用者集中声明受保护的表
├── DataScopeVisitListener # 唯一拦截点
└── BypassDataScopeAspect  # @BypassDataScope 的窄范围 AOP

infra-security             ← 认证实现（不碰数据权限）
├── SaTokenCurrentUser implements CurrentUser
├── AuthFacade (interface)
└── SaTokenAuthFacade implements AuthFacade

platform-iam.domain.auth
└── AuthService + DataScopeLoader  # 登录时查角色-数据范围规则并存入 session
```

**关键取舍**：

| 曾经考虑 | 最终选择 | 理由 |
|---------|---------|------|
| `DataScopedRepository` 基类 | **砍掉** | MyBatis-Plus 继承惯性的残留，VisitListener 已覆盖所有拦截场景 |
| `DataScopeContext` ThreadLocal 作"业务态容器" | **砍掉** | `CurrentUser` 已经是单一数据源，ThreadLocal 冗余且带异步丢失风险 |
| 让 `infra-jooq` 依赖 `infra-security` | **不放宽** | 把 `CurrentUser` 接口沉到 `mb-common` 就不需要 |
| PostgreSQL Row Level Security（RLS） | **不用** | 对 meta-build 独立开发者用户画像过重；记入 `future-capabilities.md` 作为企业级升级路径 |

## 成本和风险

### 成本

| 项目 | 成本 | 备注 |
|------|------|-----|
| 5 轮方案迭代的讨论时间 | 中（约 2 小时） | 已发生，不可逆；但过程沉淀为本 ADR，未来可省 |
| 文档修订（backend-architecture.md + CLAUDE.md） | 中（约 1 小时） | 一次性 |
| 写本 ADR | 低（约 30 分钟） | 一次性 |
| **使用者的运行时心智**：新增业务表要记得注册到 `DataScopeRegistry` | 低-中 | 由 canonical reference 示范 + `CLAUDE.md` MUST #11 强化 + code review 兜底 |

### 风险

| 风险 | 严重度 | 缓解 |
|------|-------|------|
| 使用者忘记在 `DataScopeConfig` 注册新表 → 表可以读写但数据权限不生效 | 中-高 | canonical reference 示范 + §4.5 步骤 10.1 + `CLAUDE.md` MUST #11 + 测试层面写"新表必须有数据权限集成测试"的规范 |
| 使用者用 `dsl.fetch(String)` 等原始 SQL → 绕过 VisitListener | 中 | ArchUnit `NO_RAW_SQL_FETCH` 自动拦截（基于 jOOQ `@PlainSQL` 注解匹配） |
| VisitListener 对 jOOQ 的 AST 访问逻辑 bug | 中 | M4 阶段写完整的数据权限集成测试矩阵（各 DataScopeType × 各受保护表） |
| 未来加一个新的 jOOQ 版本，VisitListener API 变了 | 低 | jOOQ VisitListener API 稳定多年，Lukas Eder 的博客和官方文档都有长期维护 |
| 使用者有场景必须跨部门查询但忘了加 `@BypassDataScope` | 低 | 业务报错时容易定位；`@BypassDataScope` 强制填 `reason`，code review 能追溯 |

### 放弃的能力

| 放弃 | 对 meta-build 的影响 |
|------|-------------------|
| 基类继承带来的"写 Repository 时感觉有父类兜底" | **这其实不是能力，是心理安慰**。方案 E 的保护更强（VisitListener 在更底层） |
| `DataScopeContext.set(specialScope)` 这种"程序性数据范围临时覆盖" | v1 不需要此场景；未来如果需要，可以在 `BypassDataScopeAspect` 的基础上扩展 `TemporaryDataScope.execute(scope, supplier)` |
| nxboot `DataScopeAspect` 的代码直接复用 | 本来就不值得搬运（见本 ADR 核心论点） |

## 维护约定

### 本 ADR 是元 ADR，未来所有"从 nxboot 借用"的决策必须遵守

每当新的 ADR 或 specs 修订涉及"从 nxboot 借用 XXX"时，**必须**在借用说明里回答本 ADR 的三个审查问题：

1. 这个组件在 nxboot 里的范式来源（哪个生态的约定）
2. meta-build 新技术栈对同一问题的原生答案
3. 改造策略（直接搬 / 部分借用 / 只借鉴概念 / 彻底放弃）

如果借用说明里没有回答这三个问题，就是"mechanical inheritance"，违反本 ADR。

### 附录 A 的每一行都是本 ADR 的审查对象

`backend-architecture.md` 附录 A 的"改造点"列应该**随时反映本 ADR 的思考过程**。如果将来发现某一行的改造点写得过于机械（"直接搬"但没说原生范式是否匹配），应该重新审视。
