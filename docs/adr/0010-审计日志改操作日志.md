# ADR-0010: v1 用操作日志替代审计日志（platform-audit → platform-oplog）

- **状态**: 已采纳
- **日期**: 2026-04-12
- **决策者**: 洋哥 + 小灵犀
- **相关文档**: [`docs/specs/backend/03-platform-modules.md §2.3 platform-oplog`](../specs/backend/03-platform-modules.md)、[`01-module-structure.md §4 Domain Model`](../specs/backend/01-module-structure.md)

> **注**：本 ADR 写于 `backend-architecture.md` 已拆分到 `docs/specs/backend/` 子目录结构之后，所有对 backend 文档的引用直接指向子目录文件。

## 背景

### 原方案：platform-audit（审计日志）

M0 初始设计中，`platform-audit` 模块使用 `@Audit` 注解，计划实现**字段级 diff 审计**——记录每次操作的 before/after 快照，支持"谁在什么时间改了哪个字段从 A 到 B"的审计追溯。

### M0 Review 中的质疑

Review 讨论中明确了两个问题：

1. **v1 不需要字段级 diff**：字段级审计需要 before/after 快照、diff 算法、审计表 schema 设计（JSON 存储 or 拆表）、查询性能优化等，复杂度远超 v1 需要
2. **命名误导**："audit"在企业软件领域通常暗示合规审计（compliance audit），包括不可篡改、保留期策略、审计报表等——v1 都不做

### 洋哥的决策

> v1 不做审计日志（字段级 diff），只做操作日志（Controller 入参/返回值）。

---

## 决策

### 功能范围

| 维度 | v1 操作日志（已实现） | v1.5+ 审计日志（推迟） |
|------|-------------------|---------------------|
| 记录粒度 | **操作级**：谁、什么时间、做了什么操作、入参、返回值 | **字段级**：哪个字段从 A 改到 B |
| 注解位置 | Controller 方法（`@OperationLog`） | Service / Repository（需要拦截数据变更） |
| 存储 | 单表 `mb_operation_log` | 可能需要多表（主表 + 变更明细表） |
| 快照 | 不做 before/after 快照 | 需要 before/after 快照 + diff 算法 |
| 复杂度 | 低（AOP 拦截 Controller 入参和返回值即可） | 高（需要 Entity 变更追踪或 Hibernate Envers 类方案） |

### 命名翻转

| 原命名 | 新命名 | 原因 |
|--------|--------|------|
| `platform-audit` | `platform-oplog` | 模块名反映实际功能（操作日志，非审计日志） |
| `@Audit` | `@OperationLog` | 注解名反映实际行为（记录操作，非审计 diff） |
| `sys_audit_log` / `mb_audit_log` | `mb_operation_log` | 表名同步（注意：表前缀也同步改为 `mb_`，见 ADR-0009） |
| `AuditService` | 无独立 Service | AOP 切面直接写表，不需要业务层 Service |

### `@OperationLog` 注解设计

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface OperationLog {
    String action();                    // 操作标识，如 "iam.user.create"
    String targetType() default "";     // 操作对象类型，如 "User"
    String targetIdExpr() default "";   // SpEL 表达式取目标 ID，如 "#result.id"
}
```

**放置位置**：必须在 Controller 方法上（与 `@RequirePermission` 同层），Service 层**禁止**导入此注解。

```java
@PostMapping
@RequirePermission("iam.user.create")
@OperationLog(action = "iam.user.create", targetType = "User", targetIdExpr = "#result.id")
public UserVo create(@RequestBody @Valid UserCreateCmd cmd) {
    return userService.create(cmd);
}
```

### AOP 切面行为

1. 拦截标注了 `@OperationLog` 的 Controller 方法
2. 记录：操作人（`CurrentUser.userId()`）、时间、action、入参 JSON、返回值 JSON、IP
3. 异步写入 `mb_operation_log` 表（`@Async`，不影响主流程性能）
4. 异常时记录失败状态和异常摘要

### v1.5+ 升级路径

如果未来需要字段级审计，可以：
1. 新建 `platform-audit` 模块（与 `platform-oplog` 并存，不是替换）
2. 在 Repository / RecordListener 层实现 before/after 快照
3. 操作日志和审计日志是两个独立能力，不互斥

## 成本与风险

| 维度 | 评估 |
|------|------|
| v1 节省的复杂度 | 高（省掉 diff 算法、快照存储、审计查询 UI 等） |
| 文档替换成本 | 低（全局替换，本次已完成） |
| 风险 | 低——操作日志覆盖"谁做了什么"的核心需求；字段级 diff 是锦上添花 |
| 放弃的能力 | v1 无法回答"某字段从 A 改到 B 是谁改的"——需要 v1.5+ 补充 |
