# ADR-0021：采用虚拟线程 + Java 25 LTS 升级 + 通知分发取消同步聚合

- **状态**：已采纳
- **日期**：2026-04-18
- **决策者**：洋哥 + 小灵犀
- **相关文档**：[`docs/specs/backend/`](../specs/backend/README.md)、ADR-0007（继承遗产前问原生哲学）、ADR-0008（一致性 > 局部优化）
- **类型**：架构翻转（推翻 commit `88081d75` 引入的统一超时 + 同步聚合设计）

---

## 背景

### 触发事件：代码审查发现 NotificationDispatcher 的线程泄漏风险

2026-04-18 多角色代码审查，性能审查员指出 `NotificationDispatcher.dispatch()` 在 commit `88081d75` 的实现里有三叠雷：

1. `CompletableFuture.orTimeout()` 只让外层 future 完成 TimeoutException，**不中断底层 mbAsyncExecutor 线程**
2. JavaMailSender 默认无 socket timeout，SMTP 黑洞可让线程 hang 几分钟
3. `DeliveryLogService` 的 `@Async` 和渠道 send 共用同一个 `mbAsyncExecutor`，队列满时死锁

本以为是"配超时参数"的问题，小灵犀给了方案：
- `MbNotificationProperties.timeout` 配每个渠道独立超时（IN_APP 2s / EMAIL 15s / WECHAT 10s）
- `NotificationChannel` 接口加 `default Duration defaultTimeout()`
- `CompletableFuture.orTimeout(duration)` 做应用层超时

### 洋哥第一次质疑：统一超时反模式

> "可是配置统一的超时，如果我有一些业务接口需要很长的连接（比如去国外的网站下载 PDF 之类的），那这个超时时间怎么可能用统一的呢？
>
> 我觉得还是要回归第一性原理，想一想我们遇到的这个问题本身是不是由我们导致的，有没有更本质的解决方法。"

### 社区最佳实践调研

按要求调研（主要参考 context7 MCP + 2024+ 年的 Spring 官方博客 + JEP 491），核心发现：

1. **`CompletableFuture.orTimeout` 设计上就不中断线程**（Nurkiewicz 2015 定性，2024 年仍未改变）——这是 Java 标准库的刻意设计，future 和线程解耦
2. **超时必须在 I/O 层做**，业界共识：`RestTemplate` 的 socket timeout / `JavaMailSender` 的 `mail.smtp.timeout` / `mail.smtp.writetimeout` 才是真正生效的超时
3. **Java 21 虚拟线程（Project Loom）** 从根源解决阻塞 I/O 线程池耗尽问题——虚拟线程挂起时归还 carrier OS 线程，百万级虚拟线程无所谓
4. **通知场景"同步并发等所有渠道返回"本身是反模式**——Spring Modulith Events + Outbox 是 2024+ 年 Spring 官方推荐的轻量异步解耦模式

### 洋哥的第二次质疑：讲清楚原始背景

> "我们这个问题的原始背景是什么？你给我讲一下。这是什么功能？在解决什么问题的情况下引入的？请整体给我介绍一下。"

### 回溯引出"同步聚合"的根本反模式

回溯 `NotificationDispatcher` 的真实调用链：

```
用户 HTTP → NoticeService.publishOne（同步业务事务）
                    ↓ AFTER_COMMIT + @Async
          NoticePublishedEventListener  ← 已经是异步
                    ↓
          NotificationDispatcher.dispatch
                    ↓ 并发 4 渠道 allOf().join()
          InAppChannel / EmailChannel / WeChatMp / WeChatMini
```

**关键事实**：
- 用户请求在 `NoticeService.publishOne` 返回时就已经看到"发布成功"
- `NotificationDispatcher` 已经在异步线程里跑，**不阻塞用户请求**
- `allOf().join()` 等所有渠道返回后，**没有任何上游消费这个聚合结果**（监听器返回 void，事件末端）

**结论**：Dispatcher 的 `allOf().join()` 是**纯粹的反模式**——把 4 个独立的、失败互不影响的、结果无人聚合的调用，人为捆绑成同步等待。捆绑的代价就是"超时值怎么配"这个伪问题。

## 第一性原理重新推导

### 真正要解决的问题

不是"超时怎么配"，是**三叠需求**：

1. **业务需求**：用户发公告后，通知要发到各渠道；各渠道失败互不影响
2. **系统需求**：阻塞 I/O（SMTP / 微信 API / 未来的海外 PDF 下载）不能拖垮共享线程池
3. **可观测需求**：每个渠道的投递状态（SUCCESS / FAILED / TIMEOUT）要能追溯

### 现方案的根源错误

- 用"同步并发等所有渠道返回"解决了**需求 1**
- 但因为同步等，需要"统一超时"防 hang → 错误地把"业务超时"和"资源保护超时"混成同一个概念
- "资源保护超时"压根不该存在于应用层——它是虚拟线程/线程模型的职责

### 正确分层

| 层 | 职责 | 正确的技术 |
|---|---|---|
| **业务层（Dispatcher）** | 派发消息到各渠道，不等 | `executor.execute()` fire-and-forget |
| **渠道层** | 单渠道 send 的业务超时（比如邮件 15s，微信 10s，海外 PDF 下载 10 分钟） | 各渠道自己配 I/O 客户端的 socket timeout |
| **线程模型层** | 阻塞 I/O 不占 OS 线程 | 虚拟线程（Project Loom） |
| **可观测层** | 每次投递结果落表 | `DeliveryLogService`（每渠道独立写） |

每一层的"超时"是各自的职责，不再共用一个 `orTimeout` 兜所有。

### Java 21 vs Java 25 的抉择

Java 21 的虚拟线程有 **`synchronized` pinning** 问题——synchronized 块内阻塞 I/O 会钉住 carrier OS 线程，虚拟线程优势失效。JEP 491 在 Java 24 修复，Java 25 LTS（2025-09 GA，现已稳定运行 7 个月）包含 JEP 491。

不升 Java → Java 21 下虚拟线程要逐个验证依赖（JavaMailSender 的 synchronized 风险显著）。
升 Java 25 → JEP 491 让 synchronized 块内阻塞也自动挂起虚拟线程，无 pinning。

## 决策

### 决策 1：NotificationDispatcher 取消同步聚合（改 fire-and-forget）

```java
// 新实现（伪代码）
public void dispatch(NotificationMessage message) {
    for (NotificationChannel channel : channels) {
        if (channel.supports(message)) {
            executor.execute(() -> invokeChannel(channel, message));
        }
    }
    // 立即返回，不等
}

private void invokeChannel(NotificationChannel channel, NotificationMessage message) {
    long start = System.currentTimeMillis();
    try {
        channel.send(message);
        deliveryLogService.record(message, channel.channelType(), SUCCESS,
                                  System.currentTimeMillis() - start, null);
    } catch (Exception e) {
        log.error("通知渠道失败 channel={} noticeId={} exceptionType={} message={}",
                  channel.channelType(), extractNoticeId(message),
                  e.getClass().getSimpleName(), e.getMessage());
        deliveryLogService.record(message, channel.channelType(), FAILED,
                                  System.currentTimeMillis() - start, e);
    }
}
```

### 决策 2：全局启用虚拟线程

```yaml
# application.yml
spring:
  threads:
    virtual:
      enabled: true
```

```java
// AsyncConfig.mbAsyncExecutor 改用虚拟线程
@Bean("mbAsyncExecutor")
public Executor mbAsyncExecutor() {
    return Executors.newVirtualThreadPerTaskExecutor();
}
```

`@Async` / `@Scheduled` / Tomcat request 处理 全部切到虚拟线程。

### 决策 3：升级 Java 21 → Java 25 LTS

- `pom.xml`：`<java.version>21</java.version>` → `25`
- `maven-enforcer-plugin`：`[21,22)` → `[25,26)`
- CI 的 `setup-java@v4`：`java-version: 21` → `25`
- 开发者本地：`brew install openjdk@25` + `JAVA_HOME` 切换
- 实测关键依赖库兼容性（见"前置验证清单"）

### 砍掉的概念

与 commit `88081d75` 对比，**全部砍掉**：

- ❌ `MbNotificationProperties.timeout.*`（统一超时配置）
- ❌ `NotificationChannel.defaultTimeout()` 接口 default 方法
- ❌ `CompletableFuture.orTimeout(duration, unit)` 调用
- ❌ `CompletableFuture.allOf(futures).join()` 聚合等待
- ❌ 单独的 Dispatcher 层"业务响应时间保护"概念

这些概念在虚拟线程 + fire-and-forget 模型下**都是不需要存在的**。

### 保留的概念

- ✅ `NotificationChannel` 接口
- ✅ `DeliveryLogService` + `mb_notification_delivery_log` 表
- ✅ 各渠道自己内部的 I/O 超时（RestTemplate connect/read timeout / `mail.smtp.timeout`）

## 理由

### 为什么取消同步聚合

- 没有消费者等聚合结果（listener 返回 void）
- 各渠道失败互不影响
- 同步聚合是人为制造的"捆绑"，制造了"统一超时"这个伪问题

### 为什么虚拟线程 > 每渠道独立 Executor（Bulkhead）

- Bulkhead 只解决"渠道互不拖累"，不解决"长耗时业务"（比如未来的海外 PDF 下载分钟级阻塞）
- 虚拟线程从根源让"线程阻塞"这件事不再是资源问题
- Spring Boot 3.2+ 官方一等支持，Java 21/25 LTS 原生能力
- 一次投资全局受益（所有 @Async、Tomcat、JDBC、RestTemplate 调用都受益）

### 为什么升 Java 25 LTS

- JEP 491 让 synchronized 阻塞不再 pin carrier thread
- Java 25 是 LTS，稳定支持到 2030+
- 2026-04 时点 Java 25 已 GA 7 个月，主流依赖均已兼容
- M5 阶段（无真实生产流量）是换 JDK 最佳时机

### 为什么不走 Outbox 模式

- Outbox（Spring Modulith Events + DB outbox 表）是通知场景的**终态正解**
- 但工程代价最大，且当前 v1 通知量没到痛点
- 虚拟线程 + fire-and-forget 已经把线程资源问题彻底解决
- Outbox 留作 v1.5+ 演进路径（若未来通知量激增需要跨 JVM 重试/审计）

## 影响

### 正面影响

1. **统一超时这个伪问题消失**：不同业务（海外 PDF 下载 vs 微信 API）各自在 I/O 客户端层配超时，互不干扰
2. **线程池耗尽问题消失**：虚拟线程轻如鸿毛，阻塞多少个都不占 OS 线程
3. **代码极简**：Dispatcher 不到 30 行，无 `CompletableFuture` 复杂链式调用
4. **未来长耗时业务天然受益**：海外 PDF 下载、对账拉取、多平台发单都不用设计新线程模型

### 代价

1. **Java 升级成本**：pom + CI + 本地开发机切换，一次性约 2-3 小时
2. **依赖 pinning 实测**：重点 JavaMailSender，若严重 pinning 需给 EmailChannel 单独池
3. **运行时可观测性变化**：虚拟线程的 thread dump 形态和传统不同（`jstack` 输出里 carrier / virtual 分层），运维需要适应

### 已知限制（本次不处理）

- Outbox 模式推迟到通知量增长后再评估
- 虚拟线程下 `ThreadLocal` 使用要更谨慎（虚拟线程数量大，泄漏更致命），但项目已有 `MdcTaskDecorator` / `SaTokenTaskDecorator` 封装，问题可控
- 在虚拟线程下，MDC / Sa-Token 的 `CompositeTaskDecorator` 仍生效（验证过）

### 放弃的能力

| 放弃 | 对 meta-build 的影响 |
|------|-------------------|
| "Dispatcher 层统一兜底超时" | 无损失——这个超时本身就不该存在（伪问题） |
| "用户能同步确认各渠道投递结果" | v1 本来就不支持（listener 是异步的），无变化 |
| "线程池大小 = 并发上限"的资源规划心智 | 虚拟线程下这个心智失效，改为"I/O 客户端层限流"思路 |

## 维护约定

### 关于超时

**铁律**：**应用层不再配任何"兜底超时"**。所有超时都下沉到 I/O 客户端层：
- HTTP 调用：`RestTemplate` / `RestClient` 的 connect/read timeout（per-Bean，必要时 per-request）
- 邮件：`mail.smtp.connectiontimeout` / `mail.smtp.timeout` / `mail.smtp.writetimeout`
- JDBC：Hikari `connectionTimeout` + DB query timeout
- 未来引入新 I/O 客户端（WebClient / Feign / gRPC），必须在客户端层配超时

### 关于线程模型

**铁律**：**业务代码不依赖线程池大小做并发控制**。所有限流/舱壁/熔断走应用层语义（如 Bucket4j、Resilience4j SemaphoreBulkhead），而非"池子大小限死并发"。

### 关于 Dispatcher 类似场景

未来若有"并发调用多个下游"的类似场景（比如推送通知到多个对接系统、批量对账拉取等），**不要重新引入 allOf().join() 模式**。要么 fire-and-forget，要么走 Outbox。

### 关于 Java 版本

- 项目锁定 Java 25 LTS 直到 2028 年（下一 LTS 为 Java 29）
- 升级 JDK 需新 ADR

### 关于 Outbox 演进

若未来通知量增长（QPS > 100 / 日量 > 100 万）或需要"跨 JVM 重试""投递审计"等能力，应启动 Outbox 改造（新 ADR）。届时 `mb_notification_delivery_log` 表可直接演进为 outbox 存储。
