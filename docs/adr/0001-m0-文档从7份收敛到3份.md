# ADR-0001: M0 文档从 7 份收敛到 3 份（最小骨架 + 增量补）

- **状态**: 已采纳
- **日期**: 2026-04-10
- **决策者**: 洋哥 + 小灵犀
- **相关文档**: `meta-build规划_v1_最终对齐.md`、`docs/specs/backend/`（原 `docs/specs/backend-architecture.md` 已拆分）

> **注**：本 ADR 写于 backend-architecture.md 拆分前。文中所有对 `backend-architecture.md` 的引用，现统一指向 `docs/specs/backend/` 子目录，导航见 [backend/README.md](../specs/backend/README.md)。

---

## 背景

`meta-build规划_v1_最终对齐.md` 里 M0 阶段列出了 "决策前必读 7 份文档 + 1 份 future-capabilities"，计划 3-5 天完成：

1. `docs/vision.md` — 项目定位和愿景
2. `docs/references/asset-inventory.md` — nxboot 参考资产盘点
3. `docs/specs/frontend-five-layer-architecture.md` — 前端五层架构
4. `docs/specs/backend-architecture.md` — 后端架构借用 + 修复方案
5. `docs/specs/canonical-reference-features.md` — notice/order/approval 设计
6. `docs/specs/project-structure.md` — 嵌套 monorepo 结构
7. `CLAUDE.md` + `AGENTS.md` — AI 契约
8. `docs/future-capabilities.md` — 未来能力储备

讨论中识别出几个问题：

1. **v1 实施必需性不均**：vision/canonical-reference/future-capabilities 对 v1 动手不是必需，是对外讲故事或后置阶段的需求
2. **内容重叠**：`project-structure` 的目录树本来就分属前/后端两份文档
3. **附录级内容独立成文**：`asset-inventory`（从 nxboot 借什么）本质是 backend-architecture 的附录
4. **文档失真风险**：写文档时的抽象容易与动手时的细节脱节，3-5 天的前置文档可能在 M1 动手时发现大量失真
5. **拖延动手时机**：对独立开发者，文档阶段越长越容易失去动能

---

## 决策

M0 只产出 **3 份核心文档**，采用"**能指导实施落地的骨架 + 增量补**"的写法。

### 3 份核心文档

| # | 文档 | 合并自 | 核心内容 |
|---|------|--------|----------|
| 1 | `docs/specs/frontend-architecture.md` | 原 #3 + #6 前端部分 | 5 层 package 职责边界、依赖方向、25 个 L1 组件清单、7 个 L2 组件清单、Theme Registry schema、前端目录树 |
| 2 | `docs/specs/backend-architecture.md` | 原 #4 + #6 后端部分 + #2 作为附录 | framework/system 分层、10+8 模块清单、4 个架构问题修复方案、Spring Modulith 包结构、ArchUnit 规则、从 nxboot 借用的资产清单 |
| 3 | `CLAUDE.md` 骨架 | 原 #7 的 CLAUDE 部分 | 索引式结构：概要 + 17 项关键决策列表 + 指向 specs/adr 的链接。不塞架构细节 |

### 写法约定

- **折中方案**：不是"先大而全"（2000 行完整文档），也不是"极简骨架"（只占位）。每份文档必须达到"**让任意开发者能照着动手**"的层次，但细节在 M1-M4 实施过程中按需增量补
- **不留悬念**：骨架文档里不允许出现"等实施时再定"这类占位
- **同步演进**：每完成一个阶段就回补对应文档章节，保证文档和代码一致
- **CLAUDE.md 索引化**：不把架构细节写进 CLAUDE.md，只写概要 + 链接到 `docs/specs/` 和 `docs/adr/`

## 砍掉的 5 份文档的去向

| 原文档 | 处理方式 | 触发时机 | 理由 |
|--------|---------|---------|------|
| `docs/vision.md` | 推迟 | M6 / M7 开源准备阶段 | v1 实施过程中内部用不着，对外讲故事是开源前的需求 |
| `docs/references/asset-inventory.md` | 合并 | M0 | 作为 `backend-architecture.md` 的附录章节"从 nxboot 借用的资产清单" |
| `docs/specs/canonical-reference-features.md` | 推迟 | M4 末 / M5 初 | notice/order/approval 的设计应该基于 M4 平台模块的实际代码，避免空想失真 |
| `docs/specs/project-structure.md` | 合并 | M0 | 目录树分别合并到 frontend-architecture.md 和 backend-architecture.md |
| `AGENTS.md` | 推迟 | M7 开源准备阶段 | 本质是 CLAUDE.md 的 <200 行摘要，没有完整版 CLAUDE.md 就没法抽摘要 |
| `docs/future-capabilities.md` | 推迟 | M4 结束阶段回顾时 | 是 "v1 不做但备忘" 的清单，适合在阶段回顾时沉淀，不适合前置 |

## 影响

### 正向
- M0 工作量从 **3-5 天压缩到 1.5-2 天**，更快进入动手阶段
- 避免"文档写完但和代码不匹配"的失真状态
- CLAUDE.md 索引式结构让 AI 契约更易维护（改架构 → 改 specs，CLAUDE.md 不动）
- 文档和代码同步演进，每完成一个里程碑就有一次小的文档回补

### 风险与缓解
- **风险**：骨架文档如果写得不够"能指导落地"，M1-M4 实施时仍可能卡住
- **缓解**：每份骨架文档写完后做一次"陌生人反向走查"——假设自己是从未接触过这个项目的开发者，能否照着文档动手搭建
- **风险**：推迟的文档到触发时机时被遗忘
- **缓解**：在本 ADR 里明确记录触发时机，M4 结束时回看本 ADR 决定是否补写

M0 完成的下班信号更新为：

- CLAUDE.md 骨架里的每一项关键决策都有指向 specs 或 ADR 的链接
