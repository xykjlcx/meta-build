# 0410 v1 规划收敛与 M0 文档瘦身

## v1 规划最终对齐文档产出
- 问题：桌面上的 `新项目规划_v1.md` 已过时，需要一份反映最新 17 项决策的 ground truth
- 过程：把前期和 GPT 的讨论 + 17 项技术决策整合到一份对齐文档
- 结论：产出 `meta-build规划_v1_最终对齐.md`（731 行），作为 v1 阶段所有决策的基线

## v1 规划整体 review
- 问题：需要对规划做一次批判性 review，找出过度工程化或定位模糊的地方
- 过程：读完规划文档后提出 6 条挑战 + 3 个待定问题
- 结论：洋哥逐条回复，明确：
  1. 不做 M2.5 中间发布，v1 就是要完整底座
  2. 千人千面 = CSS 覆盖 + App Shell 布局组合，边界明确
  3. meta-build 比 nxboot v3 "更朝前一步"，不是 fork，前端完全重构重写
  4. Spec 引擎推迟到 v1.5/v2 完全可以，8+10 模块和业务代码是 Spec 引擎的前置校验和模板来源
  5. 全家桶思路（8 system + 10 framework）就是脚手架的默认能力，不砍
  6. 商业价值：开源影响力 + 外包业务内部底座 + AI 原生协作效率

## 千人千面的承诺分级
- 问题：定制的三种工作流是否应该分级承诺
- 结论：
  - **主题级定制**：强承诺（Theme Registry + CSS 变量，覆盖 60-80% 场景）
  - **App Shell 换壳**：中等承诺（5 种布局预设 + 多品牌多租户皮肤）
  - **组件级源码定制**：提供实现机制（L1 源码在项目内，兜底路径，不承诺无痛）

## M0 文档从 7 份收敛到 3 份
- 问题：原 M0 列出 7 份决策前必读文档 + 1 份 future-capabilities，工作量 3-5 天，存在文档失真和过度前置的风险
- 过程：
  1. 按"v1 实施必需性"对 7 份文档分层
  2. 识别出 vision / canonical-reference / future-capabilities 对 v1 动手非必需
  3. project-structure 与 frontend/backend architecture 内容重叠可并入
  4. asset-inventory 是 backend 的附录级内容，不独立成文
  5. AGENTS.md 本质是 CLAUDE.md 摘要，没必要前置
- 结论：M0 只做 3 份核心文档：
  1. `docs/specs/frontend-architecture.md`
  2. `docs/specs/backend-architecture.md`
  3. `CLAUDE.md` 骨架
- 详见：`docs/adr/0001-m0-文档从7份收敛到3份.md`
- 工作量：3-5 天 → 1.5-2 天

## 文档写法约定
- 问题：大而全 vs 最小骨架加增量补，选哪个
- 结论：取折中——骨架要达到"能指导实施落地"的层次，细节在 M1-M4 过程中按需增量补，文档和代码同步演进

## CLAUDE.md 定位
- 问题：CLAUDE.md 应该写多详细，内容如何组织
- 结论：采用索引式结构——只写概要 + 17 项关键决策列表 + 指向 `docs/specs/` 和 `docs/adr/` 的链接。不把架构细节塞进 CLAUDE.md，保证改架构时改 specs 就够了，CLAUDE.md 不需要跟着变

## 建立 docs 目录结构
- 结论：在 `06-meta-build/` 下建立 docs 子目录
  - `docs/日志/` — 每日项目日志（跟 Studio 日志模板一致，MMDD 格式）
  - `docs/adr/` — 关键决策日志（和 v1 规划文档里的 `docs/adr/` 约定一致）
- 后续 `docs/specs/`、`docs/guides/`、`docs/prompts/` 等子目录按规划文档里的约定，M0 动手时按需创建
