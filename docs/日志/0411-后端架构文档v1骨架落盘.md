## 关键技术决策
- **响应格式**：采用 RFC 9457 ProblemDetail 混合方案
  - 成功 → 直接返回业务对象
  - 分页 → PageResult<T>
  - 错误 → ProblemDetail (application/problem+json) + 扩展字段 code/traceId
  - 理由：HTTP 语义对齐、AI 时代契约友好、OpenAPI 天然支持、Spring Boot 3 内置、i18n 自然集成
- **Flyway 命名**：按业务域 `V<域编号>_<序号>__<描述>.sql`（例 `V01_001__iam_user.sql`）
- **异常基类**：保留 BusinessException(code, msg) 自定义基类，响应序列化时映射为 ProblemDetail
- **Snowflake ID**：从 nxboot 直接借用 SnowflakeIdGenerator，所有表 id 用雪花算法
- **WebSocket**：推迟到 v1.5，v1 只留占位
- **Audit 自动化**：framework/audit AOP 骨架，v1 做框架，实际审计规则等 M5 业务模块再补
- **i18n**：第一天就做，MessageSource + Accept-Language LocaleResolver，集成到 ProblemDetail

## 后端模块命名调整（洋哥 review 过程中提出）
- 问题：洋哥问 `mb-framework` → `mb-infra`、`mb-system` → `mb-platform` 是否合适
- 过程：
  1. 分析 `framework` 的两个硬伤：和 Spring Framework 本体冲突、语义不精确（装的是 infra 能力不是 framework）
  2. 分析 `system` 的问题：太泛、行业隐含约定（RuoYi 生态的"system 后台"）外人看不懂、和 SystemException/`sys_` 前缀语义重叠
  3. 对比 infra/platform 的优点：DDD/Clean Architecture 术语对齐、和规划文档"平台业务模块"措辞一致、对 AI 契约友好
- 结论：
  - `mb-framework` → `mb-infra` ✅
  - `mb-system` → `mb-platform` ✅
  - `mb-common` / `mb-admin` 不动
  - Java 包名同步：`com.metabuild.framework.*` → `com.metabuild.infra.*`，`com.metabuild.system.*` → `com.metabuild.platform.*`
  - 10 个子模块 `framework-xxx` → `infra-xxx`，8 个子模块 `system-xxx` → `platform-xxx`
  - 规划文档 ground truth 不动，本次决策独立记录在 `docs/adr/0002-*.md`，后续文档统一用新命名
- 实施：
  - 写了 `docs/adr/0002-后端模块命名framework改infra-system改platform.md`
  - 对 `backend-architecture.md` 做了批量替换（共 4 批 Edit 调用，约 200 处替换）

## 大修订轮次（ADR-0003/0004/0005/0006 集成 + P0/P1 规范）

### 背景
review 过程中陆续识别出 4 个架构级决策需要翻转或扩展，洋哥同意"一次性大修订"避免反复改。

### 决策 1：去除 Spring Modulith（ADR-0003）
- 问题：Modulith 主包位置敏感（洋哥追问"未来加 order/billing 模块怎么办"暴露扩展性问题），概念复杂，AI 训练数据少，和 meta-build "给 AI 执行的契约" 核心命题冲突
- ROI 评估：Modulith 独特价值只有 @ApplicationModuleTest + Documenter 两项，都不是 v1 关键需求；其他能力 ArchUnit + Maven 能 95% 替代
- 结论：**移除 Spring Modulith**，改为 "Maven 模块隔离 + ArchUnit 双保险"（编译期 pom 白名单 + 测试期包级约束）
- Java 包保留 `com.metabuild.platform.iam` 分层命名（不扁平化）

### 决策 2：新增 mb-schema 数据库契约层（ADR-0004）
- 问题：洋哥质疑 jOOQ 生成代码放 `infra/jooq/generated` 是语义错配（基础设施层不应包含业务表映射）
- 洞察：jOOQ codegen 是 schema-level 的产物（跨表外键 join 需要全局引用），必须集中；类比前端的 `@mb/api-sdk`，后端应该有一个"数据库契约层"
- 结论：4 层 Maven → **6 层**，新增 `mb-schema`（Flyway SQL + jOOQ 生成代码）和 `mb-business`（使用者扩展位 + M5 canonical reference 容纳地）
- Flyway migration 从 `mb-admin` 搬到 `mb-schema`

### 决策 3：认证框架 Spring Security → Sa-Token + CurrentUser 门面层（ADR-0005）
- 参考：另一个会话的场景加权分析（10 个用户故事）
- 关键洞察 5 个维度：
  1. 场景加权（75% 核心用户场景 Sa-Token 胜）
  2. 气质匹配（shadcn/TanStack Router 简洁派 vs Spring Security 企业派的违和感）
  3. **AI 友好度**（Sa-Token 线性逻辑 vs Spring Security Filter 链容易在 CORS/CSRF/顺序上翻车）
  4. 切换对称性（两框架切换成本对称约 2-3 天，应该先选核心场景更好的）
  5. **CurrentUser 门面层**（业务层零感知，切换成本降到半天）
- 补充我的判断：
  - Token 模式用 **JWT mixed**（JWT 载体 + Redis 黑名单），无状态 + 可主动注销
  - 权限注解用**自定义 @RequirePermission**（不直接用 @SaCheckPermission，避免 Controller 耦合）
  - 测试用 **MockCurrentUser + @Primary Bean**（不用 mockStatic）
  - ArchUnit 规则强制业务层不依赖 `cn.dev33.satoken.*`

### 决策 4：canonical reference 质量规范（ADR-0006）
- 洋哥让我"从第一性原理审视"，暴露一个被长期忽略的问题：文档结构完备 ≠ 使用者能从零到一跑起来

- P1 五维度（M0 占位 + M4 前完善）：事务高级场景 / 事件命名规范 / 测试数据管理 / CI/CD 步骤 / 多实例约束
