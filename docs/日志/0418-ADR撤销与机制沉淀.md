# 0418 ADR 撤销与机制沉淀

## ADR-0021 / ADR-0022 撤销（主线）
- 问题：昨夜产出的两份后端 ADR（虚拟线程 + Java 25 / FK RESTRICT + jOOQ listener）经两轮对抗审查发现 9 条 🔴 事实错误和方案缺陷（Spring Boot 3.5.3 不兼容 Java 25 / PermissionChangeExecuteListener 自递归 / PostgreSQL `FOR KEY SHARE` 锁术语混淆 / Facade 删除后业务动作封装无处安放等）
- 过程：
  - Review 发现"补丁链式叠加"模式：方案 A 出问题 → 方案 B → 方案 B 又出问题 → 方案 C
  - 回到第一性原理质疑："到底有没有必要做这两个 ADR？问题本身影响真的大吗？"
  - 数据核查：PermissionService 无缓存在当前 QPS < 1 下 PG 毫无压力；NotificationDispatcher 无超时在当前通知量日均个位数 + `mbAsyncExecutor(core=4, max=8)` 队列稳定 0 下是"假设性"风险
  - 结论：都是伪问题。ADR 全撤
- 结论：
  - 撤销 ADR-0021 / ADR-0022，新立 meta-0023 作为过度工程化反面样本（和 ADR-0007 / ADR-0008 组成元方法论三元组）
  - 回退 3 个相关 commit：`fdc91251` 全部（PermissionWriteFacade 全套）+ `88081d75` 部分（Dispatcher orTimeout）+ `13bcc06b` 部分（PermissionWriteRule）
  - 保留真改进：DeliveryLog 表 + Repository + Service / Caffeine token 缓存 / 共享 RestTemplate Bean / `@ConditionalOnProperty` / HttpClientRule.NO_DIRECT_REST_TEMPLATE_CONSTRUCTION / ScheduledRule.SCHEDULED_METHOD_MUST_HAVE_SHEDLOCK
- 耗时：约 3 小时

## 小修一组（Actuator 端口 / SMTP timeout / RestTemplate 连接池 / delivery_log 字段重构 / CompositeTaskDecorator 重构 / ArchUnit FQN / 补 MDC 隔离测试）
- 问题：交接文档列了 8 项"小修"，默认每条都要做。过程中多数没过"当前阶段是否必要"自检
- 过程：
  - **Actuator 独立端口**：放 application.yml 导致 `Management-specific server address cannot be configured` 断言触发，89 个测试 context 加载失败。3 轮试错（全 profile port+address → test profile 覆空 port → 再覆空 address）都失败，最后回滚到 application-prod.yml only
  - **CompositeTaskDecorator 重构**：@Order + ObjectProvider 缓存方案引入"@Bean 自引用 chicken-and-egg"，MdcTaskDecorator 不被 ObjectProvider 收集到 → MDC 不传播到异步线程。改 `@Component` 独立定义才修好
  - **async_must_isolate_login_state_across_thread_pool_reuse**：测试设计过猛——Sa-Token `logout()` 销毁服务端 session，连带已捕获的 SaRequest 失效，抛 NotLoginException 假阳性。改成 `async_must_not_leak_mdc_across_thread_pool_reuse`（MDC only）
  - **delivery_log 字段重构**：`notice_id → business_id` / `duration_ms INT → BIGINT` / 新增 `business_type`(NOT NULL DEFAULT 'UNKNOWN') / `event_type` / `message_type`（消息格式，预留）/ `trace_id`。重生 jOOQ 后 Repository 同步
- 结论：
  - Actuator 独立端口只在 prod profile 启用，pre-prod checklist 登记"✅ 已做（仅 prod）"
  - CompositeTaskDecorator 原实现（每次扫 BeanFactory）确实低效但不影响正确性；@Order 缓存的新实现需要 @Component 配合，否则 @Bean 在 ObjectProvider 自引用下不可靠
  - `message_type`（消息格式）列暂无 set 逻辑，永远 NULL —— 刻意预留给未来 TEXT/HTML/TEMPLATE 区分，pre-prod 前可补
- 耗时：约 2 小时（含 3 轮 Actuator 试错 + 2 次 verify 大红大绿）

## openapi-v1.json 快照 500 错报调查
- 问题：`server/api-contract/openapi-v1.json` 自 commit `02037402`（18:34）起就是 500 错报 JSON，不是真实 OpenAPI schema。OpenapiSnapshotTest "通过" 但写入的是错报
- 过程（3 轮改口，反面样本）：
  - 根因 1 号假设："test profile `springdoc.api-docs.enabled: false` 导致 springdoc 不注册 controller" → 部分对但不完整
  - 根因 2 号假设："`@TestPropertySource(properties = "springdoc.api-docs.enabled=true")` 没生效" → 错，实测后来验证它一直生效
  - 根因 3 号真因："URL path 不匹配"——`application.yml` 配 `springdoc.api-docs.path=/api-docs`（不是默认 `/v3/api-docs`），测试硬编码请求 `/v3/api-docs` → `ResourceHttpRequestHandler` 降级 → `NoResourceFoundException` → `handleGeneral(Exception)` 兜底吞成 500
  - 连带发现次要 bug：`GlobalExceptionHandler.handleGeneral(Exception ex)` 把 `NoResourceFoundException` 吞成 500 违反 HTTP 语义（未定义路由该 404）
- 结论：
  - 修 OpenapiSnapshotTest：URL 改 `/api-docs`、加 `@TestPropertySource` 强开 api-docs、严格断言 HTTP 200 + Content-Type JSON + `response.openapi` 起始 "3." + `paths` 非空
  - 修 `GlobalExceptionHandler` 加 `@ExceptionHandler(NoResourceFoundException.class) → 404`（新增 `CommonErrorCodes.SYSTEM_NOT_FOUND`）
  - 新 snapshot 46KB：51 个端点 + 61 个 DTO schema + 4 个 tag
  - 按协议不 push，等前端 worktree 稳定后统一推（contract 变动会影响前端 orval）
- 反思：长链路调查不做对照实验是反模式。应该"列所有可能因 → 各做最小实验 → 排除到最后一个"，而不是"看到现象 → 给结论 → 下一个现象"
- 耗时：约 1.5 小时

## 机制沉淀（ROI 最高的产出）
- 问题：本会话反复踩 rule 已记录但没执行的坑：`maven-reactor-run-pitfall` 踩 2 次（改 infra-async / infra-exception 后忘 `-am`）、`over-engineering-trap`（新加）同会话内违反、`cross-review-residual-scan` 没主动做。AI 自律不可靠
- 过程：按"能工具化就工具化"的 ROI 排序，做了三层防御：
  - **Tier 1 工具强制**：
    - `.claude/settings.json` 的 SessionStart hook 把 `docs/rules/INDEX.md` 全量注入 AI context（不依赖 AI 主动扫）。验证：新会话不读文件就答出 18 pitfall + 4 playbook，AI 自己还说"索引预加载 + 具体规则按需 Read"
    - `server/scripts/mvn-test.sh` wrapper 硬拦 `-pl` 缺 `-am` 的命令
  - **Tier 2 模板**：
    - `docs/handoff/meta-review-template.md`：承接交接文档时每条任务答"是不是真问题"4 问后分类 🔴立即做 / 🟡登记到 pre-prod / 🔵撤销
    - `docs/rules/second-pass-review.md`（playbook）：高风险任务（≥3 commit / infra 层 / 契约变更 / ADR 撤销）主动做二轮 review，给 6 维度清单
  - **Tier 3 流程**：
    - `CLAUDE.md` / `AGENTS.md` 加 4 条维护约定：infra 层改动必须先 ADR / 调查类先汇报再执行 / 承接走 meta-review / 批次级做二轮 review
- 结论：SessionStart hook matcher 配错一次（`""` 不触发，正确值 `startup` / `resume` / `clear`），修好后验证成功。三层防御覆盖本会话踩过坑的 80%
- 耗时：约 2 小时

### 元反思
- 本会话最大的 meta 教训是 **"AI 自律在 rule 清晰的情况下仍然会失败"**。这迫使产品设计必须从"rule 存在就够"升级为"工具强制 + 模板 + 流程摩擦"三层防御
- 另一个元教训：**"承接模式下默认每条任务都要做"是比 "AI 过度工程" 更隐蔽的盲点**，因为它连"决策现场"都没有——任务已经被上个会话定下来了，这个会话只是在"执行"。meta-review 模板的价值就是把执行模式强制切回决策模式
