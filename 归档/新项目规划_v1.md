# 全新项目规划：AI 时代的可定制全栈技术底座（暂用代号 `scaffold-x`）

## Context

这是一个**全新项目**，不是 nxboot 的演进。基于 nxboot 积累的后端架构经验和前端组件抽象，结合 codex demo（nxboot-v2）验证的 shadcn 定制化前端方案，构建一个**真正实现"千人千面" UI 定制** 的 AI 时代全栈技术底座。

### 为什么是新项目而不是 nxboot v2

nxboot 的定位是"AI 时代的 RuoYi Plus"——重点在后端架构纪律和 Spec 驱动。但洋哥的真实愿景超出了这个范围：

- 不想锁死 UI 风格（RuoYi / nxboot / JHipster 都做不到真正的深度定制）
- 希望每个使用者都能通过 AI 对话把 UI 改成自己的样子
- 同时保持后端架构、权限、验证层不可动摇

**这需要一条不同的前端路线**（shadcn/Radix + 组件源码在项目内）和**一套不同的架构分层哲学**（L1/L2 是样式载体，L3 只关心业务）。基于 nxboot 改造会被已有的 antd 依赖、CLAUDE.md 包袱、历史决策牵制，不如开新项目。

### 新项目 vs nxboot 的本质差异

| 维度 | nxboot | 新项目 |
|------|--------|--------|
| **产品定位** | AI 执行的后端契约 | AI 执行的契约 + 可定制视觉层 |
| **UI 哲学** | 固定风格（飞书/antd/shadcn 三选一） | 千人千面(使用者定义) |
| **L1 组件归属** | 依赖 antd（node_modules）| 源码在项目内 |
| **主题机制** | CSS 预设 + antd adapter | Theme Registry + CSS 变量 + Tailwind |
| **使用者关系** | 使用者 = 开发者 | 使用者 = 品牌主，开发者 = 使用者 + AI |
| **Spec 引擎职责** | 生成前后端全栈代码 | 只生成 L3（后端 + 前端页面骨架） |

## 核心架构决策（已对齐）

### 决策 1：前端三层可定制架构

```
┌─────────────────────────────────────────────────┐
│  L3 Business Pages（业务页面层）                │
│  - Spec 引擎生成                                 │
│  - 完全依赖 L1/L2 组件，不碰样式                │
│  - 改 L1/L2 会自动传导到 L3（无缝切换）         │
├─────────────────────────────────────────────────┤
│  L2 Business Components（业务组件层）           │
│  - NxTable / NxDrawer / NxForm / NxFilter 等    │
│  - 使用者可定制（改结构或改默认样式）           │
│  - API 稳定，对 L3 暴露的 props 签名不可变      │
│  - 底层依赖 L1 + TanStack Table/RHF/Radix      │
├─────────────────────────────────────────────────┤
│  L1 Base UI Components（原子组件层）            │
│  - Button / Input / Select / Dialog 等 25 个   │
│  - 源码在 `client/src/components/ui/`           │
│  - shadcn/Radix + CVA 风格                      │
│  - 通过 Theme Registry + CSS 变量 + Tailwind   │
│  - 使用者可深度定制（改 CVA 变体或重写结构）   │
└─────────────────────────────────────────────────┘
             ↑
   Theme Registry（主题配置对象）
   - 颜色、圆角、字号、间距、布局度量、动画等
   - 优先修改目标，覆盖 ~60-80% 的定制场景
```

**关键原则（洋哥的北极星）：**
> Spec 只生成 L3，L1/L2 是样式载体。改 L1/L2 会自动让所有存量和新生成的 L3 页面无缝切换风格。这就是"千人千面"的实现机制。

### 决策 2：前端技术栈

- **基础库**：React 18 + TypeScript strict + Vite + TanStack Query + Zustand
- **UI 层**：Radix UI + Tailwind CSS + CVA + shadcn 风格组件（源码在项目内）
- **业务组件**：沿用老 nxboot 的 L2 职责划分（NxTable/NxDrawer/NxForm/NxFilter/NxBar/NxLoading/ApiSelect）
  - **但放弃严格 API 兼容**：按 shadcn 生态的自然形态重新设计 props，不被 antd 习惯绑架
- **表格引擎**：TanStack Table v8
- **表单引擎**：React Hook Form + Zod
- **图标**：Lucide React
- **代码质量**：Biome（替代 ESLint + Prettier）
- **测试**：Vitest + React Testing Library + MSW + Playwright（E2E）
- **i18n**：react-i18next（从第一天就做，中英文双语）

### 决策 3：主题定制机制

采用"方案 C：主题配置 + 组件源码双层定制"，参考三套现有实现：

1. **老 nxboot 的 3 套 CSS 预设**（antd.css / feishu.css / shadcn.css） — 学习 CSS 变量命名规范
2. **nxboot-v2 的 Theme Registry 对象模型** — 学习结构化主题对象（colors/radii/sizes/layout/shadows/motion/fonts）
3. **Ant Design 的 ThemeConfig 体系**（https://ant.design/docs/react/customize-theme-cn） — 学习语义 token 和组件 token 的分层

**初期维度（覆盖 80% 场景）：**
- 颜色：brand / foreground / background / border / muted / destructive / success / warning（含 hover/active 状态）
- 圆角：sm/md/lg/xl
- 字号和字体：fontSize scale + fontFamily
- 间距和密度：compact/comfortable 两种预设
- 布局度量：sidebarWidth / headerHeight / contextBarHeight 等
- 阴影：floating / modal / selected
- 动画：productive/expressive easing + 三档 duration

**逻辑上可不断扩展**——超出初期维度的定制走"改 L1 组件源码"的兜底路径。

### 决策 4：定制入口（使用者 UX）

**不做专用工具。** 使用者用自己的 AI（Claude Code / Cursor / Windsurf / 通义灵码）直接改文件，nxboot 的职责是：

- 定义清晰的**文件结构**（知道改哪里）
- 定义明确的**约束契约**（知道不能破什么）
- 提供高质量的 **CLAUDE.md / DESIGN.md** 指导 AI
- 通过**验证层**兜底（改坏了构建会红）

这样 nxboot 是"AI 可执行的技术底座"，不是"AI 产品"。降低维护成本，把 AI 能力的进化红利留给使用者。

### 决策 5：Spec 引擎职责（简化版）

**Spec 只生成业务层代码：**

- **后端**：VO / Command / Controller / Service / Repository / Flyway 迁移 / 集成测试
- **前端 L3**：types.ts / api.ts / columns.tsx / XxxList.tsx / XxxForm.tsx
- **菜单/权限**：自动往 `sys_menu` 表插行，绑定默认角色

**Spec 不碰 L1/L2：**
- L1 原子组件是"样式载体"，由使用者定制
- L2 业务组件是"抽象层"，由框架维护
- Spec 生成的 L3 代码通过 **import L2 组件 + 传递数据** 完成业务

**L3 被 regen 覆盖的问题自动解决**：因为 L3 不该被定制（定制应该发生在 L1/L2），所以 regen L3 不会丢失使用者的工作。

### 决策 6：后端借用策略

从 nxboot 借用**架构和代码**，但不保持 fork 关系：

**直接借用：**
- JDK 21 + Spring Boot 3.4 + jOOQ + Flyway + Redis + JJWT 整套技术栈
- BaseIntegrationTest（Testcontainers 单例 + 事务回滚）
- TestHelper（mockLoginUser / adminToken）
- CacheEvictSupport（事务提交后失效）
- SlowQueryListener、AsyncConfig、RateLimitInterceptor
- GlobalExceptionHandler、JooqHelper
- Flyway V1-V29 的 schema 设计（作为参考，不直接复用 SQL 文件）
- 14 个业务模块的 canonical reference（user / role / menu 等）

**借用时修复的 4 个架构问题：**

1. **jOOQ 泄漏到 Service 层**
   - nxboot 的 `RoleService` 直接注入 DSLContext 写 SQL
   - 新项目强制：Service 层禁止 `import org.jooq.*`，数据访问全部收到 Repository
   - 用 ArchUnit 规则固化

2. **跨领域 Service 穿透 Repository**
   - nxboot 的 `RoleService` 直接调用 `MenuRepository`
   - 新项目强制：跨领域访问走对方 Service，不走 Repository
   - 用 ArchUnit 规则固化

3. **DataScope 是 opt-in 模式**
   - nxboot 靠手贴 `@DataScope` 注解，忘贴即数据泄漏
   - 新项目反转为 opt-out：默认所有查询都注入数据权限条件，通过 `@BypassDataScope` 显式跳过
   - 实现路径：Repository 基类 + jOOQ VisitListener 双保险

4. **缓存全量失效**
   - nxboot 用 `@CacheEvict(allEntries = true)` 过于激进
   - 新项目一律 key 级失效，级联失效显式声明

**全新设计：**
- 从第一天就有 Spring Boot Actuator + Micrometer + 结构化日志
- 应用 Dockerfile（多阶段构建）
- ArchUnit 架构守护测试（5-8 条核心规则）
- 权限覆盖率自动检查
- SpotBugs + FindSecBugs + gitleaks 加入 CI
- Flyway 从 V1 开始，所有表预留 `tenant_id` 字段（默认 0，为多租户留空间）

## 三大风险和对冲

### 风险 1：L2 业务组件的重新设计

**问题：** 老 nxboot 的 L2 API 基于 antd 习惯，shadcn 生态的心智模型不同。强行保留 API 会"水土不服"，完全抛弃又丢失老 nxboot 的工程经验。

**对冲：**
- 保留 L2 的**职责划分和命名**（NxTable / NxDrawer / NxForm 等 7 个）
- **重新设计 props 接口**：按 shadcn + TanStack Table + RHF 的自然形态
- **保留抽象层次**：L3 通过 L2 调用，不直接调用 TanStack Table 和 RHF
- 每个 L2 组件写 ADR 记录"为什么这样设计"，方便未来演进

### 风险 2：主题系统过度复杂

**问题：** Theme Registry 试图覆盖太多维度，每次新增维度都要改 adapter 和所有 L1 组件。

**对冲：**
- 初期只做 7 个维度（颜色 / 圆角 / 字号 / 间距 / 布局 / 阴影 / 动画）
- 所有超出维度的定制**明确走组件源码兜底**，不扩张 Theme Registry
- L1 组件源码本身已经通过 CSS 变量 + CVA 最大化可定制性
- 升级策略：M6 阶段积累 3 个真实定制需求后再决定是否扩展 Theme Registry

### 风险 3：Spec 引擎职责边界

**问题：** Spec 只生成 L3 很优雅，但实际落地时可能有灰色地带（比如"某个模块需要自定义的操作列渲染"）。

**对冲：**
- 明确约定：**Spec 生成的 L3 有扩展点**（行渲染器、自定义动作、表单字段覆盖）
- 扩展点通过 "slot" 模式实现：生成的页面暴露 props 让使用者注入自定义片段
- 使用者的自定义片段写在单独的文件里（比如 `UserList.custom.tsx`），regen 时不受影响
- 复杂到无法用扩展点表达的，退化到"手写 L3 代码"——此时该模块不再受 Spec 管辖，Spec-Code 一致性检查会标记它为 "手工维护"

## 开始实施前的剩余决策点

以下是最终开工前需要洋哥拍板的问题。我会在写 7 份文档的过程中逐个提出：

1. **项目名字** — 已决定先用代号，后期再定

3. **代码仓库策略** — 单仓库 monorepo 还是 client/server 两个独立仓库
4. **初始主题预设** — 默认主题走飞书风格还是新设计一套"中性基调"
5. **L2 API 重新设计时的取舍** — 是否在 props 里加 i18n key 约定
6. **Spec Schema 的 v1 范围** — 只做单表 + 主从表 + 枚举状态机，还是直接上 ApplicationService 编排层
